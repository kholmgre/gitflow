GitFlow Workshop

Workshop on how to use Git for feature branching and running a project according to the GitFlow process.

Introduction

A core concept in Git is the concept of branches. On a personal, small-scale or fairly uncomplicated project you might not have the need to run separate branches. The case then is usually simply to push to master. But what should one do when working on big projects? The GitFlow process is one way of solving this issue.

This workshop will focus on training us in the use of the well-established GitFlow model. Successful use of the GitFlow model will allow us to keep better track of our own changes, and ultimately provide a safer workflow for the collective master branch which will henceforth only receive reviewed code. GitFlow offers some additional structure to the more general "Feature Branch Workflow", but all in all they work pretty much the same way.

Remember that Git is essentially just a logging system to determine where and in which context changes (and what these changes are) have taken place. When working on a larger project it makes sense to isolate any work you are doing from the work of others until your work is ready for review and subsequent integration into the overall, "master" copy of the project. You achieve this isolation by working in a separate branch of the project.

Because Git tracks changes in files, rather than actual files, you don't usually need multiple sets of the project files. However, what you need to do is make sure you work in the correct branch, so changes don't carry over straight away into another branch, like the develop or master branch.

Prerequisites

You should have a working understanding of how Git operates. You should also be practically able to add (stage), commit, merge, pull, fetch and push to a centralized Git repository before attempting this workshop.

Overview of commands

text goes here

  • git branch [some-branch]: Creates a new branch with the name you supply
  • git branch -d [some-branch]: Deletes the branch with the name you supply
  • git checkout [some-branch]: Switches to the branch with the name you supply
  • git checkout -b [newbranch] [oldbranch]: Creates a new branch (if it doesn't exist) from a previous branch and switches to the new branch (example usage: "git checkout -b myfeature develop")

Structure of branches

text goes here

  • master: This should always be a stable branch with write access only by someone who has review responsibilities
  • develop: This should be the default working branch
  • feature1: Any additional features or individual work should be separated into their own branches (use a descriptive name, such as "animated-menu-items")...
  • feature2: ...so that all work is easily overviewed and isolated

1. Add a new branch

Only push to the "develop" branch and your own feature branch! You can list the current branches in the project with git branch.

Switch to the development branch by running git checkout develop. You should now be on that specific branch, meaning all edits will be done only to that branch.

CLI

Adding it by entering git branch feature1 develop into the CLI, and then switching to it with git checkout feature1. Switching to the master branch and listing changes with git status reveals that the master branch has no registered changes – all in good order!

2. Create pull request

Only available via GUI

The pull request is used as a way to communicate that changes have taken place and that you wish these to be reviewed and added to the main branch (develop or master). A pull request should always contain a pertinent, meaningful description.

3. Merge a pull request

Only available via GUI

The pull request is done in Bitbucket or another GUI. In Bitbucket you will find "Pull requests" on the left-hand side panel, under the "Navigation" heading.

Valid users can "Approve" and "Unapprove" a pull request. The actual merge takes place when the "Merge" button is clicked. Make sure that "Source" and "Destination" are correct.
