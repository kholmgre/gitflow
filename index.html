GitFlow Workshop

Workshop on how to use Git (more specifically a GitFlow process) including how to do things like feature branching.

Some things to remember

Essentially Git is a logging system to determine where and in which context changes (and what these changes are) have taken place. A core concept in Git is the concept of branches. When working on a larger project it makes sense to isolate any work you are doing from the work of others until your work is ready for review and subsequent integration into the overall, "master" copy of the project. You achieve this isolation by working in a separate branch of the project.

Because Git tracks changes in files, rather than actual files, you don't usually need multiple sets of the project files. However, what you need to do is make sure you work in the correct branch, so changes don't carry over straight away into another branch, like the develop or master branch.

On a personal, small-scale or fairly uncomplicated project you might not have the need to run separate branches. The case then is usually just simply to push to master. You will likely have some experience with the basics of Git, like adding, committing, merging and pushing. This workshop will therefore focus on training us in the use of the well-established GitFlow model. Successful use of the GitFlow model will allow us to keep better track of our own changes, and ultimately provide a safer workflow for the collective master branch which will henceforth only receive reviewed code.

Structure of branches

  • master – This should always be a stable branch with write access only by someone who has review responsibilities
  • develop – This should be the default working branch
  • feature1 – Any additional features or individual work should be separated into their own branches (use a descriptive name!)...
  • feature2 – ...so that all work is easily overviewed and isolated

Work only with a "develop" branch

You can list the current branches in the project with "git branch".

Adding a new branch called "feature1"

Adding it by entering "git branch feature1" into the CLI, and then switching to it with "git checkout feature1".

Switching to the master branch and listing changes with "git status" reveals that the master branch has no registered changes – all in good order!
